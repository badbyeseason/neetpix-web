## 1. 市场全景图

### 1.1 产品分类

数据提取工具有三个层级，越往下 ARPU 越高、开发越重：

`
轻量 ← —————————— → 重量

[Chrome Extension]     [桌面可视化工具]        [API 服务]
    Web Scraper           Octoparse            ScrapingBee
    Data Miner            ParseHub             ScrapingFish
    Instant Scraper       Helium Scraper       Apify
    Scraper               Portia               ZenRows
    低价/高频              中价/中频             高价/低频
    C端为主                B端为主               B端为主
`

**关键判断**：Chrome Extension 是切入成本最低的层级，也是转化漏斗的入口。
站稳后可以向上延伸到桌面版或 API 服务。

### 1.2 竞品观察：Chrome Extension 侧（8 个）

扩展名 | 用户数(约) | 评分 | 定价模式 | 核心短板
-------|-----------|------|---------|---------
Web Scraper | 1,000,000+ | 3.8/5 | 免费 + 捐赠 | 动态页面支持差、选择器门槛高
Data Miner | 500,000+ | 3.7/5 | 免费(500页/月) → /月 | 模板没人维护、分页经常断
Instant Data Scraper | 700,000+ | 4.0/5 | 免费 | 批量能力弱、不支持登录页
Scraper | 400,000+ | 3.9/5 | 免费 → /月 | 导出格式少、不智能
Simple Scraper | 200,000+ | 4.1/5 | 免费(2000行) → /月 | 功能偏基础、不适合复杂页面
Data Scraper - Easy Web Scraping | 300,000+ | 3.8/5 | 免费 → /月 | 稳定性差、经常 crash
Octoparse (有 Chrome 版) | 100,000+ | 3.6/5 | 免费(10页) → /月 | 太贵、桌面版附加
Helium Scraper Extension | 50,000+ | 3.5/5 | 付费 {实价} | 界面老旧、更新慢

**空白提炼：**
- 没有一款在 "智能 + 低价 + 稳定" 三角上都做好的
- 免费版限制大的产品（Data Miner、Octoparse）用户抱怨最多
- 评分最高的 Simple Scraper（4.1）功能最简单——说明用户愿意为"简单好用"买单
- 没有一个产品做 **中英双语** 界面（中文用户需要切英文界面使用）

### 3.2 推荐的定价模型

`
免费版（获客漏斗入口）
  ├─ 每月 1,000 行数据
  ├─ CSV 导出
  └─ 基础自动检测

Pro 版 ¥48/月（66 元）
  ├─ 每月 50,000 行数据
  ├─ JSON / Google Sheets 导出
  ├─ 数据清洗规则
  ├─ 批量 URL 爬取
  └─ 优先支持

Business ¥128/月（约 180 元）
  ├─ 每月 500,000 行数据
  ├─ 定时自动调度
  ├─ 登录会话管理
  ├─ 团队协作
  └─ 专属客服
`

### 3.3 收入目标分解

| 层级 | 价格 | 月付 | 所需用户数 | 月收入 |
|------|------|------|-----------|--------|
| Pro | ¥48/月 (¥66) | ~ | 350 人 | ¥16,800 |
| Business | ¥128/月 (¥180) | ~ | 100 人 | ¥12,800 |
| **合计** | | | **450 付费用户** | **¥29,600** |

**如果定价偏保守，用户结构不同：**

| 场景 | Pro 占比 | Business 占比 | 用户数 | 月收入 |
|------|---------|--------------|--------|--------|
| 乐观(高付费率) | 60% (¥48) | 40% (¥128) | 350 人 | ¥28,000 |
| 中性(混合) | 70% (¥48) | 30% (¥128) | 450 人 | ¥32,400 |
| 保守(低价主导) | 85% (¥30) | 15% (¥80) | 600 人 | ¥22,500 |

**关键结论：** 450 个付费用户（免费→付费转化率 3-5%，需要 9,000-15,000 MAU）可达 30K/月目标。
这个量级在一个细分 B 端市场是完全可行的。

### 3.4 支付方案

- **国际用户** → Stripe / Paddle（自动处理 VAT 和全球支付）
- **中国用户** → 接入 Lemonsqueezy 或 Paddle 国际版（支持支付宝）
  - 不需要国内支付牌照
  - 不需要国内公司主体
  - 用户用支付宝扫码支付，钱直接结到你的 Stripe 账户

## 4. 技术方案与 MVP 范围

### 4.1 技术选型建议

`
┌──────────────────────────┐
│   Chrome Extension       │
│  (TypeScript + React)    │
│        ↕                 │
│   Chrome Extension API   │
│  (tabs, storage, fetch)  │
│        ↕                 │
│   Content Script         │  ← 核心：注入到页面进行交互
│  (DOM 操作 + 用户选择器)   │
│        ↕                 │
│   Background Service     │
│  (爬取调度 + 数据处理)     │
└──────────────────────────┘
`

**核心难点：动态页面渲染**

方案对比：

| 方案 | 能力 | 复杂度 | 费用 | 推荐 |
|------|------|--------|------|------|
| 纯浏览器 API (fetch + DOMParser) | 仅 HTML | 低 | 免费 | MVP 可用 |
| Headless Browser (Puppeteer/CDP) | 完整 JS 渲染 | 中 | 免费(客户端) | v1 接入 |
| WebContainer + Puppeteer | 云端完整渲染 | 高 | 有服务器成本 | v2 考虑 |

**MVP 技术策略：**
1. 先用 document.querySelector + XPath 做基础选择器
2. 对动态页面使用 Chrome CDP（Chrome DevTools Protocol）抓取渲染后的 DOM
3. 数据清洗用简单的正则 + schema 匹配，不上 ML

### 4.2 MVP 范围（4 周）

**Must Have（第 1-2 周）：**
- [x] 点击页面上元素 → 自动生成选择器路径
- [x] 自动检测列表/表格结构（识别重复 DOM 模式）
- [x] 手动调整选区 + 预览提取结果
- [x] 单页面爬取执行
- [x] 导出 CSV
- [x] 中英双语界面

**Must Have（第 3-4 周）：**
- [x] 批量 URL 爬取（导入 CSV/手动输入 URL 列表）
- [x] 分页爬取（自动识别 "下一页" 按钮）
- [x] 导出 JSON + Google Sheets
- [x] 基础数据清洗（去重、修剪空格、数字格式化）
- [x] 免费版/Pro 版功能切割 + 支付接入

**v1.1 迭代（第 5-6 周）：**
- [x] 选择器智能修复（页面结构变化后自动重匹配）
- [x] 模板存储 + 共享（社区模板市场）
- [x] 定时自动爬取调度

**v1.2 迭代（第 7-8 周）：**
- [x] 登录态支持（用户提供 Cookie/API Key）
- [x] 增量爬取（仅获取新增数据）
- [x] XPath + CSS 选择器切换

### 4.3 技术栈推荐（快速开发）

| 层级 | 推荐 | 理由 |
|------|------|------|
| 框架 | Plasmo (Chrome Extension 框架) | 比纯 MV3 快 3x，支持 React + TypeScript 开箱即用 |
| UI | React + Tailwind CSS | 生态成熟，开发快 |
| 状态管理 | Zustand | 轻量，适合扩展场景 |
| i18n | i18next | 标准方案，中英双语无痛 |
| 支付 | Paddle / Lemonsqueezy | 无需公司主体，支持支付宝 |
| 后端(如果有) | Cloudflare Workers 或 Vercel Edge | 全球低延迟，香港节点覆盖国内用户 |
| 存储 | Chrome Storage Sync + IndexedDB | 本地优先，无需服务器 |

## 5. 获客渠道规划

Chrome 扩展的获客不需要广告预算（至少初期）:

| 渠道 | 策略 | 预期效果 |
|------|------|---------|
| Chrome Web Store SEO | 标题/描述关键词优化 → 自然搜索流量 | 长期主力渠道 |
| 知乎/公众号 | 写"用 Chrome 扩展自动收集数据"类教程 | 吸引中文用户 |
| Reddit (r/webscraping, r/chrome_extensions) | 在相关社区分享实用技巧（非广告） | 吸引英文用户 |
| Product Hunt | MVP 阶段发布 | 早期种子用户 + PR |
| 竞品长尾词 | 用户在 Google 搜"web scraper 替代品"时找到你 | SEO 长尾 |
| 模板市场 | 预置热门网站的爬取模板（电商、招聘、房产） | SEO 落地 + 即用价值 |

**关键获客假设：** 大部分用户通过 "Chrome Web Store 搜索 + Google SEO" 发现新扩展。
所以 SEO 标题要覆盖高频搜索词：

> "Smart Web Scraper - 智能网页数据提取工具，支持中英文，导出 CSV/Google Sheets"

## 6. 风险分析与缓解

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|---------|
| Google 对爬取类扩展政策收紧 | 中 | 致命 | 不违反 Chrome Store 政策（用户主动触发爬取，非自动化攻击） |
| 动态页面处理不够好 | 高 | 中 | MVP 先用 CDP 方案，v2 引入云端渲染 |
| 获客不及预期 | 中 | 高 | 尽早做 landing page 测试，如果 CTM 低则 pivor |
| 竞品降价跟进 | 低 | 中 | 靠差异化功能（数据清洗、双语、Google Sheets 直出）而非价格竞争 |
| 免费版滥用 | 中 | 低 | API key + 限流 + 合理免费额度 |
| 维护模板的成本 | 中 | 中 | 模板开源 + 社区维护模式，像 Homebrew |
